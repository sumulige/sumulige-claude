# Linus 编程哲学

> 审核代码时以此为准则

## 核心原则

| 原则 | 一句话 | 检查方式 |
|------|--------|----------|
| **好品味** | 消除特殊情况，而非处理它 | 有 if/else？重新设计数据结构 |
| **不破坏** | 向后兼容是铁律 | 现有功能全部正常？ |
| **实用主义** | 解决真问题，拒绝臆想 | 生产环境真的需要？ |
| **极简** | ≤3 层缩进，函数只做一件事 | 能砍掉一半代码吗？ |

## 思考流程

每次改动前问三个问题：

```
1. 这是真问题还是臆想？ → 拒绝过度设计
2. 有更简单的方法吗？   → 永远找最简方案
3. 会破坏什么吗？       → 列出所有受影响项
```

## 代码审查输出格式

```
【品味】🟢好 / 🟡凑合 / 🔴垃圾

【问题】
- [最糟糕的部分]

【改法】
- "这个特殊情况可以消除"
- "10行→3行"
- "数据结构应该是..."
```

## 决策标准

**值得做**：
1. 先简化数据结构
2. 消除所有特殊情况
3. 用最笨但最清晰的方式
4. 零破坏

**不值得做**：
"这是在解决不存在的问题。真正的问题是 [X]。"

## 代码风格

- 函数：短小，一件事
- 命名：斯巴达式，直白精准
- 缩进：≤3 层，否则重构
- 分支：if/else 是坏味道的信号
